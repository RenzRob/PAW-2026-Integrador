Facundo Introducción

¿Qué construimos?

🎮 Juego multijugador en tiempo real

UNO Argentino para 2–4 jugadores, con reglas propias: Comodín Roba Tres, acumulación de penalidades y sistema de puntaje global.

🤖 Bots con Inteligencia Artificial

Jugadores virtuales potenciados por Gemini 1.5 Flash que razonan su jugada a partir del estado del juego.

Desplegado en Railway y en una VM de Google Cloud Platform, con dominio real, HTTPS y pipeline CI/CD automático.

Historial de partidas y puntajes en MySQL 8. Partidas activas en memoria para máxima velocidad.

Flujo JWT

1. POST /api/ingresar → servidor genera JWT
2. JWT se guarda como cookie HttpOnly
3. Cada request lleva la cookie automáticamente
4. Middleware requireAuth valida el token
5. WebSocket upgrade también valida el JWT de la cookie

✓ Contraseña hasheada con bcrypt (10 rounds) — nunca se almacena en texto plano.

Renzo Frontend / IA

Bots con Inteligencia Artificial

🧠 Gemini 1.5 Flash

Cuando le toca el turno a un bot, el servidor envía a la API de Gemini el estado completo del juego: su mano, la carta en mesa, la penalidad acumulada y la cantidad de cartas de los rivales.

📋 Respuesta estructurada

Gemini devuelve un JSON con { cartaId, colorElegido }. El servidor valida que la jugada sea legal antes de ejecutarla — el bot respeta exactamente las mismas reglas que un jugador humano.

🛡️ Fallback robusto

Si la API falla o responde algo inválido → fallback determinista: se juega una carta de acción válida aleatoria (o número si no hay). La partida nunca se traba.

⏱️ Experiencia natural

Delay de 1.2 segundos entre turnos de bot para que se sienta como un jugador real. Hasta 3 bots por partida (Bot-A, Bot-B, Bot-C).

Gonzalo Infraestructura

Plataformas de deploy

🚂 Railway (PaaS)

Deploy desde Dockerfile

Railway detecta el Dockerfile del repositorio y construye la imagen automáticamente. Sin gestionar VMs ni sistemas operativos.

Variables de entorno integradas

Los secrets (DB, JWT, Gemini API key) se configuran directamente en el panel de Railway. Provisión de MySQL como servicio gestionado.

Dominio y HTTPS automáticos

Railway genera un subdominio con HTTPS listo para usar sin configurar Nginx ni Certbot manualmente.

☁️ Google Cloud Platform (IaaS)

Compute Engine (VM Debian)

Control total sobre el entorno. IP estática reservada para que el dominio Duck DNS no cambie entre reinicios.

Firewall + Nginx propio

Reglas GCP que habilitan solo :80 y :443. Nginx corre dentro del Docker Compose y termina TLS con Let's Encrypt.

CI/CD via GitHub Actions + SSH

Push a main → GitHub Actions se conecta a la VM por SSH, hace git pull y reinicia los containers.

🔑 Certificados TLS

Let's Encrypt emite certificados gratuitos. Certbot los renueva automáticamente cada 12 h. TLS 1.2 y 1.3, cifrado ECDHE.

📡 WebSocket cifrado (WSS)

Nginx proxea /ws con cabeceras Upgrade y Connection: upgrade. El WS queda cifrado como WSS — mismo certificado.

Headers de seguridad:   Strict-Transport-Security (HSTS)  ·  X-Content-Type-Options: nosniff  ·  X-Frame-Options: SAMEORIGIN  ·  Referrer-Policy

Gonzalo CI/CD

Pipeline CI/CD — GitHub Actions

push → main
trigger del workflow
GitHub Actions se activa
①
SSH a la VM  — con la clave privada almacenada como GitHub Secret
②
git pull origin main  — obtiene el último código
③
Genera .env en la VM inyectando los secrets (DB, JWT, Gemini, dominio)
④
docker compose --profile https up -d --build app nginx
⑤
Certbot renueva cert si es necesario → nginx reload

El código fuente se monta como volumen en el container → no se reconstruye la imagen en cada deploy, sólo se reinicia el proceso.
